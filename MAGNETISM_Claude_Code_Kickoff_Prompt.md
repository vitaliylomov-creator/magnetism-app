Ти починаєш новий білд: **MAGNETISM**. Це не черговий feature-запит — це новий standalone застосунок, окремий репозиторій, окремий Supabase-проєкт. Перш ніж написати перший рядок коду, прочитай три файли в цій директорії, у такому порядку:

1. **`MAGNETISM_Project_Definition.md`** — ЩО і НАВІЩО. Продуктова філософія, голос, non-negotiables, комерційна логіка.
2. **`MAGNETISM_Technical_Brief_Claude_Code.md`** — ЯК. Стек, файлова архітектура, database schema, edge function flow, спринт-план.
3. **`PRINCIPAL_OS.md`** — живий приклад одного з двох готових документів "корпусу мудрості" (wisdom corpus). Це не довідковий матеріал для тебе особисто — це контент, який піде в базу `wisdom_corpus` (Технічний бриф, Розділ 4–5) як частина статичного корпусу в Sprint 1.

## Одне речення, яке не можна забути

**MAGNETISM — це щоденник, який відповідає, не дашборд і не turn-by-turn чат.**

Формула (Project Definition, Розділ 4.1 / Technical Brief, Розділ 8):

> Юзер пише завершену думку → ніхто інший не відкриє цю сторінку → голос не перебиває, поки юзер не закінчив → тоді дає відповідь по суті.

Практичний наслідок для тебе як розробника: ніякого потокового чату з "typing indicator" одразу після відправки, ніяких лічильників сесій чи progress bar на головному екрані, ніякої gamification. Юзер пише повний запис, натискає один submit, і отримує одну цілісну відповідь — як лист у відповідь на лист. Якщо в якийсь момент реалізація зʼїжджає в бік звичайного чат-інтерфейсу — зупинись і звірся з Розділом 8 Технічного брифу, це регресія, не варіація дизайну.

## Голос — інший регістр, ніж MATE/MUSE/NORTH

MAGNETISM не використовує Edge OS voice (race engineer, гострота, публічний тон). Тон MAGNETISM — приватний, теплий, прямий, без сюсюкання. Деталі — Project Definition, Розділ 4.

## Non-negotiables, які не обговорюються (Project Definition, Розділ 7)

- Safety gate — обов'язковий крок ПЕРЕД будь-якою відповіддю LLM, не feature-флаг, не опція для пізніше. Sprint 2 у спринт-плані існує саме для цього і не рухається назад у пріоритеті.
- Ніякої діагностики, ніяких клінічних ярликів.
- При ознаках гострої кризи — миттєвий, теплий, конкретний redirect на реальну допомогу. Мовчання чи спроба "розібратись у чаті" — заборонені.
- Окремий Supabase-проєкт. Не переюзати проєкт MATE — психологічні дані ізольовані фізично, не тільки логічно.
- Дизайн проти залежності: без нескінченного скролу, без вимірювання успіху хвилинами в застосунку.

## Перше завдання: тільки Sprint 1

Дивись Технічний бриф, Розділ 10 (Sprint Plan) і Розділ 3 (File Architecture). Sprint 1 — це:

- новий Supabase-проєкт (auth, database)
- базовий чат-flow через Claude API (Supabase Edge Function)
- обидва OS-документи (Magnetism OS — уточни в мене, якщо файл відсутній у цій директорії; Principal OS — вже є) як **статичний** корпус у системному промпті, без RAG і без embeddings поки що (RAG — Sprint 3, не раніше)
- файлова структура фронтенду точно за Розділом 3 Технічного брифу — не перейменовуй і не реорганізовуй файли без потреби

Не переходь до Sprint 2+ без мого явного підтвердження, навіть якщо Sprint 1 виглядає завершеним раніше очікуваного.
