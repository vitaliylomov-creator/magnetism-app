# MAGNETISM — Технічний документ для Claude Code

**Статус:** ТЗ для реалізації. Читати після `MAGNETISM_Project_Definition.md` — цей документ описує ЯК будувати, той — ЩО і НАВІЩО.
**Форма:** новий standalone застосунок, окремий репозиторій, окремий домен. Не модуль всередині talent-mates-pddr.
**Стек навмисно копіює вже перевірений стек MATE AI** (той самий засновник, той самий Supabase-акаунт, менше сюрпризів при білді) — з відхиленнями, позначеними явно там, де MAGNETISM цього вимагає.

**Продуктова метафора (обов'язкова до прочитання перед кодом):** MAGNETISM — це щоденник, який відповідає, не чат-дашборд. Деталі й наслідки для UX — Розділ 4.1 `MAGNETISM_Project_Definition.md` і Розділ 8 цього документа. Якщо реалізація зʼїжджає в бік звичайного чат-інтерфейсу з typing-індикаторами й лічильниками сесій — це регресія, не варіація.

## 3. FILE ARCHITECTURE

### Frontend
```
magnetism-app/
├── magnetism-redirect.html     ← Entry point (3s loading, як у MATE)
├── magnetism-auth.html         ← Sign in / Create account + опційний link до MATE/MUSE/NORTH
├── magnetism-onboarding.html   ← Профіль: домен (MATE/MUSE/NORTH/жоден), перше знайомство, база патернів
├── magnetism-dashboard.html    ← Основний екран запису (щоденник, не дашборд — див. Розділ 8) + ritual check-in UI
├── magnetism-consultants.html  ← Каталог живих консультантів, booking flow
└── index.html                  ← Landing page
```

**Примітка щодо назви `magnetism-dashboard.html`:** назва файлу лишається технічною (узгоджена з рештою неймінгу проєкту), але UX цього екрана — не панель метрик. Перше, що бачить користувач, — чиста сторінка для запису, ближча до аркуша паперу, ніж до продуктового дашборду. Не перейменовувати файл — уточнити тільки в коментарі коду й UX, щоб наступний розробник не почав будувати графіки й лічильники за замовчуванням.

### Supabase Edge Functions
```
supabase/functions/
├── magnetism-chat/index.ts       ← CORE: safety-gate → wisdom corpus RAG → voice assembly → memory
├── magnetism-memory-refresh/index.ts  ← Cron: раз на 6 місяців, пересумовує профіль
├── create-checkout/index.ts      ← Stripe checkout (підписка)
├── stripe-webhook/index.ts       ← Активація підписки
└── consultant-booking/index.ts   ← Booking + Stripe one-off charge за консультацію
```

## 4. DATABASE SCHEMA (ключові таблиці)

| Таблиця | Призначення |
|---|---|
| `users` | Профіль: email, лінк до MATE/MUSE/NORTH account_id (nullable), домен інтересу |
| `memory_profiles` | Куровані патерни користувача — оновлюється кроном раз на 6 місяців, не на кожному повідомленні |
| `conversations` | Історія чату — role, content, timestamp, flagged_by_safety_gate (bool) |
| `wisdom_corpus` | Чанки OS-документів (Magnetism OS, Principal OS, + майбутні) з embeddings, тегами модуля й домену |
| `safety_incidents` | Лог кожного спрацювання redirect-рефлексу — timestamp, anonymized context, чи показано ресурс |
| `consultants` | Список фахівців, домен (спорт/культура/бізнес), доступність |
| `consultant_bookings` | Записи на консультацію, статус оплати |
| `subscriptions` | Stripe-статус, period_end |

**`memory_profiles` — критична деталь:** це НЕ сирий лог повідомлень. Це структурований підсумок ("юзер повторно уникає ситуацій X", "патерн Y активний"), який LLM генерує раз на 6 місяців з історії, а не читає всю історію щоразу. Так вирішена і продуктивність, і рішення "оновлюється кожні 6 місяців" з Розділу 9 Project Definition.

**Кнопка очищення пам'яті:** DELETE на `memory_profiles` + `conversations` для юзера, з explicit confirm modal ("ти втратиш всю історію патернів, ця дія незворотна").

## 5. WISDOM CORPUS — RAG-ШАР

- Кожен OS-документ (Magnetism OS, Principal OS, майбутні) розбивається на чанки по модулях, не механічно по символах — модуль Cabane залишається цілим чанком, не розрізаний навпіл.
- Кожен чанк має теги: `document`, `module`, `domain` (universal / mate / muse / north).
- При запиті користувача: embed запит → top-K пошук у `wisdom_corpus` (K=3-5) → відібрані чанки йдуть у системний промпт як контекст, а не весь документ.
- Це відрізняє MAGNETISM від "LLM відповідає з голови" — відповідь завжди прив'язана до конкретного кураторого джерела.

## 6. MAGNETISM-CHAT EDGE FUNCTION FLOW

```
1. Receive: { message, user_id, session_id }
2. Verify Supabase auth token
3. SAFETY GATE (перше, до всього іншого):
   a. Класифікація повідомлення на ознаки гострої кризи
   b. Якщо flagged → миттєвий redirect-response (кризові ресурси), 
      лог у safety_incidents, STOP — не йти далі по пайплайну
   c. Якщо не flagged → продовжити
4. Fetch user profile + memory_profile (не повна історія, підсумок)
5. Визначити домен користувача (mate/muse/north/none) для overlay-контексту
6. RAG-пошук у wisdom_corpus (top 3-5 релевантних чанків)
7. Build system prompt:
   voice persona base → memory profile → domain overlay → wisdom corpus chunks → 
   ritual/check-in state → мова відповіді
8. Call Claude API
9. Save до conversations (з прапорцем safety-gate статусу)
10. Return { response, ritual_prompt_if_any }
```

**Крок 3 не опціональний і не може бути вимкнений фіче-флагом.** Це єдиний рефлекс, узгоджений у Розділі 7 Project Definition — якщо його немає в MVP, продукт не готовий до launch, незалежно від того, скільки іншого готово.

**Крок 7 — де саме живе метафора щоденника:** система не відповідає в реальному часі на кожне слово. Вона чекає, поки юзер надішле завершений запис (одна кнопка "Готово" / submit, не потоковий чат), і тільки тоді відповідає одним цілісним повідомленням. Немає "typing indicator", що зʼявляється одразу — відповідь приходить як лист у відповідь на лист, не як репліка в діалозі.

---

## 7. SAFETY GATE — ДЕТАЛІ РЕАЛІЗАЦІЇ

- Легкий класифікатор (промпт до Claude з вузькою інструкцією "тільки визнач ознаки гострої кризи, нічого більше") запускається окремим викликом ПЕРЕД основним response-викликом — не одним promt'ом з усім іншим.
- При спрацюванні: фіксована відповідь-шаблон з контактом кризової лінії (локалізована під мову юзера), без спроби "розібратись у чаті".
- Кожне спрацювання логується в `safety_incidents` для рев'ю під час тестування живими консультантами (Розділ 7 Project Definition).
- Living-consultant тестувальники перед launch: прогнати набір edge-case повідомлень через safety gate, підтвердити, що redirect спрацьовує коректно й не спрацьовує хибно на звичайні повідомлення про стрес чи невдачу.

---

## 8. RITUAL / CHECK-IN ШАР — ЩОДЕННИК, НЕ ДАШБОРД

**Продуктова метафора, знайдена й зафіксована під час копірайтингу лендінгу (липень 2026), — ядро цього розділу:** ти пишеш → ніхто інший не відкриє цю сторінку → голос не перебиває, поки ти не закінчив → тоді дає відповідь по суті. Спортсмену чи музиканту потрібно виговоритись так, щоб хтось слухав, не перебивав, і потім дав пораду — саме це закриває MAGNETISM, і саме це має відчуватись з першого екрана.

**Наслідки для реалізації:**

- Дашборд не відкривається як "нескінченний чат за замовчуванням" і не відкривається як панель метрик. Перший екран — короткий check-in у форматі запису: як справи з ціллю тижня, який домен зараз у фокусі, вільне поле для тексту.
- **Немає лічильників сесій, стріків чи progress bar на головному екрані.** Це порушувало б Розділ 3 Project Definition ("Дизайн проти залежності") і саму метафору щоденника — щоденник не гейміфікує ведення записів.
- **Взаємодія — не turn-by-turn.** Юзер пише повний запис і надсилає його одним submit, а не обмінюється короткими репліками. UI не показує "typing..." одразу після відправки — пауза перед відповіддю є частиною відчуття "хтось справді прочитав", не багом, який треба приховати спінером.
- Після check-in — юзер може перейти в вільний запис, але дефолтний UX веде через ритуал, не повз нього.
- Візуально — ближче до чистого аркуша чи особистого кабінету (Розділ 11), ніж до продуктової панелі з картками й графіками.

## 9. LIVE CONSULTANTS — BOOKING FLOW

- `magnetism-consultants.html`: каталог фахівців із доменом (спорт/культура/бізнес), коротким біо, ціною консультації.
- Booking: вибір слоту (Calendly embed простіше на старті, ніж власний календар) → Stripe one-off charge → підтвердження.
- Ці ж консультанти — перші тестувальники safety-gate перед публічним launch.

## 11. DESIGN SYSTEM — ЧЕРНЕТКА

Навмисно НЕ копіює Edge OS (фіолетовий, блейд-діагональ, race-engineer). MAGNETISM — приватний реєстр, інший тон:

| Токен | Напрямок |
|---|---|
| Палітра | тепліші нейтральні (кремовий, графіт), один спокійний акцент — не фіолетовий Edge OS |
| Типографіка | той самий підхід — чиста, без клатеру, але менш "технічна", більш editorial |
| Композиція | більше negative space, менше "дашборд", більше "щоденник/кабінет" |

**Уточнення (липень 2026):** "щоденник, більше не дашборд" — це не тільки композиційний напрямок, а продуктова інсайт-фраза, знайдена в сесії копірайтингу лендінгу: *"The diary that writes back."* Вона визначає не тільки візуальну мову, а й саму інтеракційну модель — див. Розділ 8 цього документа й Розділ 4.1 Project Definition. Будь-яка майбутня дизайн-робота над MAGNETISM має перевірятись питанням: *це ще відчувається як особистий запис, чи вже сповзло в продуктову панель?*

Це чернетка для розробки, не фінальний бренд-гайд — фінальне рішення про візуальну мову має пройти через NORTH/Brand Consulting процес окремо, як зафіксовано в правилі неперетину ліній.

## 13. НЕ РОБИТИ

- Не запускати Sprint 3+ без завершеного й протестованого Sprint 2 (safety gate).
- Не переюзати Supabase-проєкт MATE для економії часу — ізоляція даних з Розділу 7 не опціональна.
- Не давати LLM відповідати без RAG-контексту з wisdom_corpus "про людське око, поки RAG не готовий" — краще статичний корпус (Sprint 1) за замовчуванням, ніж відповідь з хмари загальних знань.
- Не вмикати публічний launch до Sprint 7 (валідація живими консультантами).
- **Не будувати головний екран як turn-by-turn чат чи дашборд з метриками.** Це порушує продуктову метафору щоденника (Розділ 8) — регресія, не варіація дизайну.
